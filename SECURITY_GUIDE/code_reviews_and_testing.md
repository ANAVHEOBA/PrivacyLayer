# Code Reviews, Testing, and Release Readiness

PrivacyLayer should treat code review as a safety mechanism, not just a social
formality. A reviewer is not there to approve effort; the reviewer is there to
intercept bugs that could cost users money or privacy.

## 1. What Reviewers Should Prioritize

Reviewers should look for:

- broken invariants around deposits, withdrawals, roots, and nullifiers
- implicit trust expansions in admin or verifier configuration
- serialization mismatches between circuits, contracts, and future clients
- privacy leaks in logs, metadata, or frontend defaults
- tests that only prove the happy path

A good review comment is specific and falsifiable. Examples:

- "This change lets admin update verifier parameters without a visible audit
  event."
- "This circuit input order must stay aligned with the verifier contract."
- "This UI text overstates what privacy guarantees the protocol gives."

## 2. Testing Expectations

The repository should not rely on a single test style.

Recommended layers:

- **Unit tests** for storage, validation, and small state transitions
- **Integration tests** for end-to-end deposit/withdraw flows
- **Negative tests** for malformed proofs, invalid roots, duplicate nullifiers,
  unauthorized admin calls, and paused-state behavior
- **Documentation checks** for security docs that define operational procedure

For every security-relevant PR, ask:

1. What breaks if this code is wrong?
2. Which test proves that failure mode is blocked?
3. Which negative test proves the unsafe path still fails?

## 3. Review Checklist for Sensitive Changes

Use this checklist before approving changes to contracts, circuits, or security
documentation:

- Are all new assumptions written down?
- Does the PR widen admin authority, even indirectly?
- Does any new config mutate proof or verifier behavior?
- Do tests cover both success and failure cases?
- Does the documentation avoid promising perfect privacy?
- Is rollback or incident handling affected?

If any answer is unclear, the PR is not ready yet.

## 4. Documentation as a Security Control

In security-heavy projects, documentation is part of the control surface.
Missing or misleading docs cause real incidents.

Examples:

- If note backup procedures are not documented, users lose funds by operator
  error rather than code error.
- If pause behavior is undocumented, users continue risky actions during an
  incident.
- If release notes do not call out verifier changes, auditors review the wrong
  assumptions.

That means docs PRs still deserve review rigor. "It is only documentation" is
not a sufficient reason to skip scrutiny.

## 5. Testing Before Merge

Before merging a high-risk change, maintainers should know:

- what was tested locally
- what was tested in CI
- what was not tested and why

Honest testing notes are better than vague claims like "tested against repo
structure." A good verification section names commands, files, and limits.

Examples of useful testing notes:

- `cargo test -p privacy_pool`
- `nargo test` for the affected circuit package
- snapshot tests updated intentionally for a known reason
- manual review of admin-only code paths if a toolchain is unavailable

## 6. Release Readiness Questions

Before release, ask the uncomfortable questions:

- If this change is wrong, can users lose funds?
- If this change is wrong, can users lose privacy silently?
- If this change is wrong, can maintainers still pause or recover safely?
- If this change is right, have we explained it well enough for auditors?

If the project cannot answer those questions clearly, the release is not ready.

## 7. Human Review Habits That Help

Strong habits:

- reviewing small diffs instead of giant bundles
- asking for exact invariants, not generic reassurance
- tracing a sensitive value from input to storage to output
- checking negative paths, not just successful examples
- reading the issue scope and acceptance criteria before approving

Weak habits:

- approving because the author sounds confident
- focusing on formatting while skipping state transitions
- trusting autogenerated docs or summaries without opening the files
- accepting security claims with no reference to the actual code path

## 8. Final Maintainer Checklist

Before merge:

- the scope matches the issue
- the docs are specific and not placeholder text
- the review history is clean and not padded with spammy follow-ups
- the verification section is honest
- payout and claim metadata are present if the bounty workflow expects them

If maintainers follow that checklist consistently, they reduce both technical
risk and the softer process risk that comes from unclear, rushed, or
over-automated submissions.
